# C360客户全景平台 - 业务逻辑闭环与模块审查报告 (PC & H5)

基于《Customer360项目需求规格说明书》及目前 PC/H5 端的模块开发情况，系统在基础数据展示上已初具雏形，但**部分核心业务流程中出现严重断点，导致业务流无法形成闭环**。

按照用户的要求，以下审查问题已根据 **严重级别 (Severity)** 进行重构标注，并且所有“未能形成业务闭环”的核心断点已统一整理置于本报告最头部，**同时为每个问题精准定位并附上了《PRD 需求文档》中的对应原条款要求**：

---

## 🛑 【核心】业务未闭环断点清单（按严重程度降序）

这些问题直接导致系统只能用作“只读数据大屏”，无法完成 PRD 中要求的 SCRM（社交化客户关系管理）闭环动作，必须**高优先级解决**以打通系统命脉：

### 1️⃣ 商机管理：前端核心系统入口“黑盒”化，状态未展现反馈
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：PC 端存在 `leadManagement` 代码物理目录，但在路由及界面中完全未配置展现操作入口。
- **未闭环痛点**：商机是本系统的最核心业务产出出口。前端无法管理商机（如配置计算规则、查看推送列表），也就无法形成（新建 -> 待推送 -> 已下发 -> BDC跟进中 -> 成功转化/流失）的营销全状态流转闭环，导致核心业务等同于空转、“黑盒”化。
- **📄 对应 PRD 需求文档条款**：
  > **4.9.1. 商机规则配置功能**：支持业务用户基于标签管理、顾客分群管理模块自定义商机；支持用户基于规则的商机推送配置规则。
  > **4.9.4. 商机生命周期与结果闭环功能**：支持查看商机推进情况；系统完整记录商机从生成到闭环的全流程数据，包括状态变更记录、跟进备注等，为业务分析提供数据支撑。
- **填补建议 (CRUD)**：
  - **增/查/改/删 (CRUD 全套要求)**：需要整体点亮商机页面。包括【商机计算规则管理 (C/R/U/D)】和【商机状态追踪列表 (R/U)】。

### 2️⃣ 沟通记录与跟进留痕：销售触达后动作无法回写
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：H5 的 `CommunicationRecords.vue` 与 PC 详情页的 `interactions` 均只有单向的只读展示时间轴。
- **未闭环痛点**：缺乏用户录入“写”数据的业务闭环。一线销售或 BDC 呼叫/面谈客户后，无法在前端录入“每一次的沟通结果（如：首次电访、接通且高意向客户、空号）”。导致任何商机的生命周期只能长期停摆，业务成果和客户意向无法回溯至平台。
- **📄 对应 PRD 需求文档条款**：
  > **4.9.4. 商机生命周期与结果闭环功能**：商机跟进结果由BDC和DMS等源数据同步至本系统；系统完整记录商机从生成到闭环的全流程数据。
  > **2.2.1. 商机生成 / 3.4. 自动化商机生成**：商机由C360自动生成后推送给BDC跟进，根据返场具体项目闭环商机结果。
- **填补建议 (CRUD)**：
  - **增 (C)**：H5与PC端必须明确补充【新增跟进活动记录】表单区域或全局悬浮按钮，要求必填“沟通方式”、“沟通结果枚举（继续跟进/转化/战败）”及“跟进备注”。
  - **改 (U)**：坐席提交后，由前端发请求必定连带更新该条客户身上相关商机或任务的 `Status`（完成事件闭环）。

### 3️⃣ 分群管理：完成圈选过滤后无法执行最终触达推送
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：PC端 `segmentManage` 分群列表中，操作列只支持“预览”和“导出CSV文件”。
- **未闭环痛点**：业务方花大力气圈出的人群分类，目前无法按照 PRD 要求一键【绑定派发为营销任务事件】。整个“客群划分”功能停滞在了“数据看盘跑批”这一阶段，无法直连系统的外呼行动指令。
- **📄 对应 PRD 需求文档条款**：
  > **4.8. 顾客分群管理模块 & 4.9. 商机管理模块**：商机专注分发，基于系统内的标签和分群进行商机规则配置功能。
  > **5.2.1. 商机推送接口**：向 BDC 系统推送标准化客群数据、客群主数据。
- **填补建议 (CRUD)**：
  - **触发 (Action)**：针对状态 `Active` 的分群记录，补充极关键的【推送到商机/营销触达】操作按钮，点击后拉起指令弹窗配置要推送的沟通渠道（外呼/短信/企微）并触发服务端投递队列。

### 4️⃣ 顾客视图与数据回流机制：一线纠错无操作接口
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：H5 和 PC 端的数据档案面板绝大多数为只读看板。
- **未闭环痛点**：PRD 要求必须要为 BDC 客服留有“赋予和剥除手工标签”、“提交联系方式错误或用户死亡等客户异常”的业务功能。由于前后端目前不具备完善的双向表单体系，系统缺乏获取数据一线人员反馈机制的反哺能力。
- **📄 对应 PRD 需求文档条款**：
  > **4.11. BDC嵌入页和Wecom嵌入应用**：数据交互：支持 BDC/WeCom 端数据回传（标签添加或删除、顾客信息准确性反馈）。
  > **4.7.3. 标签反馈与更正流程**：允许用户通过前端对识别出的标签不一致或错误提交反馈 / 更正。
- **填补建议 (CRUD)**：
  - **增 (C)**：H5 端增加全局【纠错与修改反馈提交】快捷方式入口，只要提交，此记录必须自动进入 PC 端的 `errorCorrection`（异常控制中心）待处理待办流转。
  - **增/删 (C/D)**：两端均需全面放开支持【手工属性标签】的池内下拉拉取点击赋予或剥夺权限。

### 5️⃣ 档案管理与手工上传：业务可用性断层 (数据上传后未形成流转闭环)
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：系统已开发完整的 `dataQualityWorkbench` (数据质量工作台) 进行多源 (C@P, POAS, WWS) 数据预上传检查，也能通过表单进行核对校验，但无法将拦截出的异常数据行项推送导向给业务处理人员。
- **未闭环痛点**：按 PRD (4.1.1) 指出，该全景图需要接收极多例如 C@P, POAS, Voucher 等线下活动的依赖手动上传对接数据。这部分异常数据如果仅在技术层报错，不能直接投递给一线审核修补者，等于离线数据源仍然处于孤岛和漏斗流失中。
- **📄 对应 PRD 需求文档条款**：
  > **4.1.1. 多源数据接入**：明确包含 C@P, POAS, Voucher, WWS 以及 Manual Files 的 Connect Type 为 `Manual upload to C360`。
  > **4.1.3. Manual upload数据的时效性保障机制**：要求系统支持上传工作，并有数据校验失败/未上传的追踪与邮件提醒机制。
- **填补建议 (CRUD)**：
  - **连/改 (Link/U)**：需要在 `dataQualityWorkbench` 的预检异常结果弹窗流中补充异常流转派发，直接实现“导入不合格抛出 -> 连线至异常人工校验控制中心 (`errorCorrection`) -> 人工修补属性 -> 正式落库”的连贯审查链路闭环。

### 6️⃣ 审批规则中心：架构硬编码，未对多数关键动作进行包裹审核
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：目前的 `approval` 组件代码仅服务限于 OneID “多条记录合并覆盖”的审批情境。
- **未闭环痛点**：PRD 要求一切可能带来负面高危破坏的举措（如：新建高权敏特征标签、极广范围的客群数据明细包导出请求）都理应走系统人工拦截判定。当前该工具模块无法闭环拦截系统各方风险。
- **📄 对应 PRD 需求文档条款**：
  > **4.7.2. 标签生命周期管理**：包括标签创建、属性修改、版本控制、废弃停用等操作，确保标签体系的规范性。
  > **3.1 统一顾客识别 (OneID)**：通过规则匹配、算法融合、人工复核的分级方式... 系统自动匹配关联记录，推送数据管理员人工确认后合并。
- **填补建议 (CRUD)**：
  - **查/改 (R/U)**：在页面开发层面，抽屉(Drawer)内容的对比详细展示部分应被拆离、重新组件插槽化设计架构。务必增配 `ObjectType`（对象判断类型，需包纳 IdentityMerge, TagCreate, SegmentExport 等业务范畴的审核能力展示支撑）。

## 🟡 其他模块级 CRUD 功能健全与交互体验优化（次要改进项）

此类问题并非彻底阻断日常系统运作核心脉络的症结点，但为了平台的体验规范与稳健依然需要进行改善：

### 7️⃣ 异常合并控制中心 (Error Correction) 缺少操作动作“后悔药”
- **严重级别**：🟡 **[中 / Medium]**
- **问题点**：业务方如果在处理冲突解决列表时出现由于判定失误发生的错误性 OneID 对象的强制融合确认操作，事发后整个界面不存在补救或者恢复旧资料的退回指令。
- **📄 对应 PRD 需求文档条款**：
  > **4.5.2. 解决机制**：管理员完成操作后，系统留存修改和审计轨迹... 支持人工调整操作还原或追溯记录。
  > **3.1. 统一顾客识别**：异常场景（关键信息变更）：自动追溯历史身份记录... 可选择更新关联OneID，或创建新 OneID。
- **填补建议 (R/D)**：
  - **反向/撤 (R/D)**：必须在处理界面中独立出来另一个 Tab 专门针对【人历史主导操作台账查阅】，提供并关联【重回初始状态撤销】的回滚命令流转能力。

### 8️⃣ 数据商业看板：偏袒侧重于 IT 与系统运行基准，忽略实际客户商业 ROI 转化
- **严重级别**：🟡 **[中 / Medium]**
- **问题点**：仪表盘现阶段为“Data Monitor（系统检测运行仪表）”。然而站在商业业务产出的视角，营销部分人员更追求他们发送出去跟进短信或者推线的商机实际接听拨通率等。业务反馈果效环节存在看板断层。
- **📄 对应 PRD 需求文档条款**：
  > **2.2.3. 分析类场景**：如过去一段时间参加活动产值、哪些顾客曾经被BDC/WeCom成功触达，哪些未能成功触达（存在较高流失风险）。
  > **2.2.4. 基础指标类场景**：需支持计算并展现诸如“顾客增长”、“留存与渗透率”、“新转续/续保率”、“活动参数覆盖率”、“增换购比例”等核心指标分析。
- **填补建议 (R)**：
  - **查 (R)**：重磅建立一个新的菜单节点进入【业务核心成果看盘】，使用 Echarts 漏斗图分析并大屏追踪由“新立项 -> 完成派发BDC -> 有效回看 -> 已落成收款”的转化比率数值统计展现。

### 9️⃣ 全局标签设定上下架全生命周期缺乏“防误删”屏蔽安全规范
- **严重级别**：🟢 **[低 / Low]**
- **问题点**：部分在业务初期部署建立好的分类标签词条如果发生不再适合后续营销、而被运营方利用单纯彻底的“硬级联删除”进行下线清库，将极大程度引起过往已挂着此类词汇触发历史规则商机的追索链断损而抛出运行时报错。
- **📄 对应 PRD 需求文档条款**：
  > **4.7.2. 标签生命周期管理 (创建、修改、版本、废弃)**：允许授权业务用户基于标签运算进行灵活配置，并支持标签全生命周期管理：包括标签创建、属性修改、版本控制、废弃停用等操作。标签定义修改并且启用后，历史数据不进行自动重跑。
- **填补建议 (R/U/D)**：
  - **查 (R)**：强区分且增添字段为表示“起草 / 正进行待复查认证 / 合法发布 / 过时且失效”四种系统应用生命体标识类状态。
  - **软下线撤除 (U -> D)**：在点击弃置此种数据行记录指令动作发生并被后台接收后，仅应当以“不合法隐写逻辑进行失效标记控制（软删除）”，通过接口强力阻挡下发于全站各层各类筛选下拉列表中并剥夺新对象应用权的方式维护其过去使用的持久不报错展示。
