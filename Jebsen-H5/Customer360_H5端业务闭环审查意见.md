# C360客户全景平台 - 业务逻辑闭环与模块审查报告 (H5端专区)

基于《Customer360项目需求规格说明书》及目前 H5 端的模块开发情况，以下是专属 **H5前端 (WeCom/BDC 嵌入端)** 的业务未闭环断点清单，并包含对 **设计合理性** 与 **信息/字段齐全性** 的核对结论。

**范围说明**：资产（卡券）模块**不设计二维码及核销功能**，仅做卡券列表与状态查看；核销由其他系统或线下流程承担。

### 是否应改为多 Tab 页面（多路由）？— 结论与建议

**结论：当前单页 + 内嵌 Tab 的设计在嵌入场景下可用，但从长期可维护性、深链与分享、以及 BDC/一线使用心智来看，更推荐演进为「多 Tab 页面（多路由）」结构。**

| 对比维度 | 当前：单页 + 内嵌 Tab + 弹窗 | 推荐：多 Tab 页面（多路由） |
| :--- | :--- | :--- |
| **心智模型** | 所有内容都在「一个客户页」里，靠 Tab 和弹窗区分。 | 顶部或底部 Tab 对应「画像 / 车辆 / 沟通 / 业务」等明确页面，每个 Tab 一个路由。 |
| **深链与分享** | 无法直接打开「某客户的沟通记录」或「车辆详情」链接。 | 可支持 `/customer/:id/communications`、`/customer/:id/vehicles` 等，企微/短信可带链接直达。 |
| **维护成本** | `Home.vue` 体积过大，所有 Tab 与弹窗逻辑集中，难以按模块拆人维护。 | 每条路由对应一个视图文件（如 `Communications.vue`、`Vehicles.vue`），职责清晰、便于按模块迭代。 |
| **性能** | 首屏需承载画像 + 多个 Tab 的组件引用（即使用 v-if 按 Tab 渲染）。 | 可配合路由懒加载，仅加载当前 Tab 对应 chunk，首屏更轻。 |
| **浏览器行为** | 切换 Tab 不会改变 URL，返回键无法「返回上一 Tab」。 | Tab 即路由，返回键语义明确，与用户预期一致。 |

**推荐的多 Tab 页面结构示意**

- **Tab1 - 画像（首页）**  
  路由：`/` 或 `/customer/:oneId`  
  内容：OneID 切换、基础属性卡片、标签、商机入口、资产入口、经办人（公司）、底部可保留「业务明细」子 Tab（维保/保险/活动/金融）或收起到「更多」。
- **Tab2 - 车辆**  
  路由：`/customer/:oneId/vehicles` 或 `/vehicles`（若 oneId 从 store 取）  
  内容：车辆列表 + 车辆详情（当前 `VehicleInfo.vue` 填空后的内容）。
- **Tab3 - 沟通记录**  
  路由：`/customer/:oneId/communications`  
  内容：沟通记录列表 + **新增跟进记录** 按钮/表单（闭环写回）。
- **Tab4 - 业务明细（可选）**  
  路由：`/customer/:oneId/business`  
  内容：维保 / 保险 / 线下活动 / 金融贷款 的子 Tab，或拆成四个平级 Tab 视产品优先级而定。
- **资产 / 商机**  
  可继续用弹窗从首页或各 Tab 唤起；资产模块不设计二维码及核销功能，无需独立核销页。

**实施建议**

- 若近期以「补齐闭环功能」为主，可先保持单页，在现有 Tab 内完成沟通记录新增、商机状态回写、车辆详情等。
- 在规划下一阶段或重构时，**优先引入多路由**：先增加路由与懒加载，把「画像 / 车辆 / 沟通 / 业务」拆成独立页面，再统一底部或顶部 Tab 导航与路由同步（如用 `router-link` 或 `vue-router` 的 `activeClass` 与当前 path 绑定）。这样既保留「一个客户」的上下文（store 不变），又得到多 Tab 页面的可维护性与可扩展性。

## 🛑 【核心】业务未闭环断点清单


### 2️⃣ 顾客视图与数据回流机制：一线纠错无操作接口
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：H5 的数据档案面板绝大多数为只读看板。
- **未闭环痛点**：PRD 要求必须要为 BDC 客服留有“赋予和剥除手工标签”、“提交联系方式错误或用户死亡等客户异常”的业务功能。由于前后端目前不具备完善的双向表单体系，系统缺乏获取数据一线人员反馈机制的反哺能力。
- **📄 对应 PRD 需求文档条款**：
  > **4.11. BDC嵌入页和Wecom嵌入应用**：数据交互：支持 BDC/WeCom 端数据回传（标签添加或删除、顾客信息准确性反馈）。
  > **4.7.3. 标签反馈与更正流程**：允许用户通过前端对识别出的标签不一致或错误提交反馈 / 更正。
- **填补建议 (CRUD)**：
  - **增 (C)**：H5 端增加全局【纠错与修改反馈提交】快捷方式入口，只要提交，此记录必须自动进入 PC 端的 `errorCorrection`（异常控制中心）待处理待办流转。
  - **增/删 (C/D)**：需全面放开支持【手工属性标签】的池内下拉拉取点击赋予或剥夺权限。

### 3️⃣ 车辆全生命周期管理：视图组件完全空白
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：`src/views/VehicleInfo.vue` 文件已创建但大小为 0 字节（内容完全空白），未实现任何 UI 或交互。
- **未闭环痛点**：销售与售后人员完全无法在 H5 端查看客户名下的车辆详情、合同信息及全生命周期状态。
- **📄 对应 PRD 需求文档条款**：
  > **2.3 车辆全生命周期 (Vehicle Life Cycle — Full List)**：记录车辆从销售线索、合同签署到售后保有全过程的精细字段。
- **填补建议 (CRUD)**：
  - **查 (R)**：尽快完成 `VehicleInfo.vue` 视图的页面搭建，对接后端的车辆详情接口。
  - **改 (U)**：预留对车辆运行状态（或解绑操作）的反馈入口。

### 4️⃣ 资产与卡券精细化模块：以账户权益形式呈现，仅做查看
- **严重级别**：🟢 **[已满足需求]**
- **当前状态**：资产已在 `Home.vue` / `PortraitView.vue` 内通过**账户权益信息卡片 + 资产信息弹窗**实现，展示卡券列表、面额、状态、有效期等；`Asset` 类型与 `getAssets` 接口齐全，无独立 `Asset.vue` 路由。
- **范围说明**：**账户权益/资产模块不设计二维码及核销功能**，仅提供卡券列表与状态查看，核销由其他系统或线下流程承担。
- **📄 对应 PRD 需求文档条款**：
  > **2.4 资产与卡券精细化 (Assets & Coupons — Full List)**：覆盖所有营销权益、售后返券及购车礼包（15 项精细化字段）。
- **填补建议 (CRUD)**：
  - **查 (R)**：当前弹窗已满足「查看卡券」需求，无需增加二维码/核销能力；若需深链可仅为「查看更多」增加独立路由页（仍不包含核销）。

### 5️⃣ 公司账户经办人管理：多联系人维护无接口
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：`Home.vue` 中能展示基础的经办人/联系人信息，但缺乏经办人的 CRUD（增删改查）管理入口。
- **未闭环痛点**：B 端企业客户的人员变动频繁，销售在拜访时如发现企业联系人变更，无法在手机端实时更新系统记录，导致下次联系依然找错人。
- **📄 对应 PRD 需求文档条款**：
  > **2.8 公司账户经办人管理 (B-Side Handler Management)**：针对公司账户，需独立维护多位联系人信息（包含角色、优先级管理）。
- **填补建议 (CRUD)**：
  - **增/改/删 (C/U/D)**：当 H5 处于 `Company` 视图且拥有权限时，允许业务员通过弹窗管理（新增/停用）该企业的经办人、送修人和主要联系人。

## 🔧 建议优先级与接口/类型缺口清单

**建议开发优先级（与断点对应，按最新约束修订）**  
1. **P0**：`VehicleInfo.vue` 页面实现（现已补齐），对接车辆详情/列表接口，展示 PRD 2.3 全生命周期字段。
2. **P0**：`VehicleInfo.vue` 页面实现（现已补齐），对接车辆详情/列表接口，展示 PRD 2.3 全生命周期字段。  
3. **P0**：电话号码管理闭环——通过 `MobileEditor` 支持电话号码新增/编辑/删除、主号切换与关系标签维护（当前已实现，需与后端接口联调验证）。  
4. **P1**：顾客纠错与标签——全局纠错入口、手工标签赋予/剥夺（含接口与权限）。  
5. **P1**：经办人 CUD（公司视图下经办人增/改/停用接口与弹窗管理 UI）。

**接口/类型缺口速查（按最新设计修订）**  
- 沟通记录：已存在 `CommunicationRecord` 类型，前端通过 `GET /customer/communication-records` 只读获取，**不再需要 `POST /customer/communication-records` 新增接口**；沟通流水由源系统 T+1 回传。  
- 商机：由上游平台生成与状态流转，H5 做商机列表与状态的只读展示，已增加 T+1 同步提示引导。
- 电话号码：已存在 `MobileData` / `MobileItem` 类型及 `MobileEditor` 组件，需联调 `addMobileItem` / `updateMobileItem` / `deleteMobileItem` 接口，确保电话号码可在 H5 端完整管理。  
- 经办人：缺 `POST/PUT/DELETE /customer/handlers`（或等效）及前端管理弹窗。  
- 车辆：已有 `getVehicles`、`updateVehicleStatus`，车辆详情页 UI 已实现，需根据后端字段持续对齐优化。
