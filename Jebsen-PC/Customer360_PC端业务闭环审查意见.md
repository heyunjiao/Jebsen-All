# C360客户全景平台 - 业务逻辑闭环与模块审查报告 (PC端专区)

基于《Customer360项目需求规格说明书》及目前 PC 端的模块开发情况，以下是专属 **PC管理端** 的业务开发遗漏与未闭环断点清单：

---

## 🛑 【核心】业务未闭环断点清单（按严重程度降序）

### 1️⃣ 商机管理：前端核心系统入口“黑盒”化，状态未展现反馈
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：PC 端存在 `leadManagement` 代码物理目录，但在路由及界面中完全未配置展现操作入口。
- **未闭环痛点**：商机是本系统的最核心业务产出出口。前端无法管理商机（如配置计算规则、查看推送列表），也就无法形成营销全状态流转闭环，导致核心业务等同于空转、“黑盒”化。
- **📄 对应 PRD 需求文档条款**：
  > **4.9.1. 商机规则配置功能**：支持业务用户基于标签管理、顾客分群管理模块自定义商机；支持用户基于规则的商机推送配置规则。
- **填补建议 (CRUD)**：
  - **增/查/改/删 (CRUD 全套要求)**：需要整体点亮商机页面。包括【商机计算规则管理 (C/R/U/D)】和【商机状态追踪列表 (R/U)】。

### 2️⃣ 沟通记录与跟进留痕：销售触达后动作无法回写
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：PC 详情页的 `interactions` 只有单向的只读展示时间轴。
- **📄 对应 PRD 需求文档条款**：
  > **4.9.4. 商机生命周期与结果闭环功能**：商机跟进结果由BDC和DMS等源数据同步至本系统；系统完整记录商机从生成到闭环的全流程数据。
- **填补建议 (CRUD)**：
  - **增 (C)**：在 Profile360 的“沟通记录”Tab头部加入【新建沟通 (Log Interaction)】按钮，允许坐席在 PC 端电开系统呼叫完毕后手工补录记录更新商机状态。

### 3️⃣ 分群管理：完成圈选过滤后无法执行最终触达推送
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：PC端 `segmentManage` 分群列表中，操作列只支持“预览”和“导出CSV文件”。
- **未闭环痛点**：业务方花大力气圈出的人群分类，目前无法按照 PRD 要求一键【绑定派发为营销任务事件】。
- **📄 对应 PRD 需求文档条款**：
  > **4.8. 顾客分群管理模块 & 4.9. 商机管理模块**：商机专注分发，基于系统内的标签和分群进行商机规则配置功能。
- **填补建议 (CRUD)**：
  - **触发 (Action)**：针对状态 `Active` 的分群记录，补充极关键的【推送到商机/营销触达】操作按钮，拉起指令弹窗触发服务端投递队列。

### 4️⃣ 档案管理与手工上传：业务可用性断层 (数据上传后未形成流转闭环)
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：系统已开发完整的 `dataQualityWorkbench` (数据质量工作台) 进行多源 (C@P, POAS, WWS) 数据上传与预检，但无法将其拦截的异常数据导向给业务处理人员。
- **📄 对应 PRD 需求文档条款**：
  > **4.1.1. 多源数据接入**：明确包含 C@P, POAS, Voucher, WWS 以及 Manual Files 的 Connect Type 为 `Manual upload to C360`。
- **填补建议 (CRUD)**：
  - **连/改 (Link/U)**：需要在 `dataQualityWorkbench` 的预检异常结果流中补充异常流转派发，直接实现“导入不合格抛出 -> 连线至异常人工校验控制中心 (`errorCorrection`) -> 人工修补属性 -> 正式落库”的连贯审查链路闭环。

### 5️⃣ 审批规则中心：架构硬编码，未对多数关键动作进行包裹审核
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：目前的 `approval` 组件代码仅服务限于 OneID “多条记录合并覆盖”的审批情境。
- **📄 对应 PRD 需求文档条款**：
  > **4.7.2. 标签生命周期管理**：包括标签创建、属性修改、版本控制、废弃停用等操作，确保标签体系的规范性。
- **填补建议 (CRUD)**：
  - **查/改 (R/U)**：务必增配 `ObjectType`（对象判断类型，需包纳 IdentityMerge, TagCreate, SegmentExport 等业务范畴的审核能力展示支撑）。

## 🟡 其他模块级 CRUD 功能健全与交互体验优化（次要改进项）

### 6️⃣ 异常合并控制中心 (Error Correction) 缺少操作动作“后悔药”
- **严重级别**：🟡 **[中 / Medium]**
- **填补建议 (R/D)**：必须在处理界面中独立出来另一个 Tab 专门针对【人历史主导操作台账查阅】，提供并关联【重回初始状态撤销】的回滚命令流转能力。

### 7️⃣ 数据商业看板：偏袒侧重于 IT 与系统运行基准，忽略实际客户商业 ROI 转化
- **严重级别**：🟡 **[中 / Medium]**
- **填补建议 (R)**：建立一个新的菜单节点进入【业务核心成果看盘】，追踪由“新立项 -> 完成派发BDC -> 有效回看 -> 已落成收款”的转化比率数值统计展现。

### 8️⃣ 全局标签设定上下架全生命周期缺乏“防误删”屏蔽安全规范
- **严重级别**：🟢 **[低 / Low]**
- **填补建议 (R/U/D)**：仅应当以“不合法隐写逻辑进行失效标记控制（软删除）”，强力阻挡下发于全站各层各类筛选下拉列表中。

## 📄 用户界面逐页分析矩阵 (Page-by-Page Analysis)

基于各页面实际代码文件，提炼其实际承载职责与缺漏建议摘要：

| 目录/页面视图 | 页面主要职责与当前承载能力 | 业务完整度分析与改进建议 |
| :--- | :--- | :--- |
| `dataQualityWorkbench` | **数据融合入口**。提供向多个周边系统集（DMS、POAS、WWS等）数据补录和手工Excel数据导入预检的控制台与审计查询列表。 | **功能良好，缺链路延展**。页面完成度很高，附有合规检查反馈。建议对接 `errorCorrection` 让不可直接导入数据流转化以人工处理。 |
| `segmentManage` | **用户分群列表大盘**。查看通过打签或条件圈出的人群分类，支持查看数量、状态并导出CSV。 | **缺乏行动触角**。业务圈选人群是为了营销动作。必须追加“一键派发至外呼/发券任务队列”或“推送至企微/商机”入口按钮。 |
| `customerSegmentation` | **分群组配漏斗工作台**。核心操作区拥有极度完善的多维复杂条件交叉过滤器和决策树预估模型评估面板。 | **能力极为惊艳**。前端已支持极为繁杂合规的校验及图谱反馈，现阶段仅缺少对于创建后直接绑卡触发下发的功能，后续按此继续迭代即可。 |
| `approval` (审批中心) | **系统合规管控区**。供权限管理员针对部分敏感信息核查（目前硬编码只有记录合并申请）。 | **未被复用延展**。需改造为动态插槽审批台，让标签新建、活动导客等系统所有高危动作均流入此台审批。 |
| `dashboard` | **运行大盘视窗**。观测系统中每天导入数据多少，清洗异常比例情况。 | **业务指标失明**。过度倾向技术监控，需大面积引进“商机转换漏斗分析图”、“客流跟进有效沟通率”等更贴合管理层诉求的模块展示。 |
| `customerList -> Profile360` | **台账单客透视卡**。极为详尽列明此单客的各种名下明细关联业务与属性点。 | **只能看不能“做”**。在互通记录 Tab 中，应该设立“主动沟通回写”框以直接向服务单内增加记录完成跟进回路闭环。 |
