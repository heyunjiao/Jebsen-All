# C360客户全景平台 - 业务逻辑闭环与模块审查报告 (PC端专区)

基于《Customer360项目需求规格说明书》及目前 PC 端的模块开发情况，以下是专属 **PC管理端** 的业务开发遗漏与未闭环断点清单：

---

## 🛑 【核心】业务未闭环断点清单（按严重程度降序）

### 1️⃣ 商机管理：前端核心系统入口“黑盒”化，状态未展现反馈
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：PC 端存在 `leadManagement` 代码物理目录，但在路由及界面中完全未配置展现操作入口。
- **未闭环痛点**：商机是本系统的最核心业务产出出口。前端无法管理商机（如配置计算规则、查看推送列表），也就无法形成营销全状态流转闭环，导致核心业务等同于空转、“黑盒”化。
- **📄 对应 PRD 需求文档条款**：
  > **4.9.1. 商机规则配置功能**：支持业务用户基于标签管理、顾客分群管理模块自定义商机；支持用户基于规则的商机推送配置规则。
- **填补建议 (CRUD)**：
  - **增/查/改/删 (CRUD 全套要求)**：需要整体点亮商机页面。包括【商机计算规则管理 (C/R/U/D)】和【商机状态追踪列表 (R/U)】。


### 3️⃣ 分群管理：完成圈选过滤后无法执行最终触达推送
- **严重级别**：🔴 **[最高 / Critical]**
- **当前状态**：PC端 `segmentManage` 分群列表中，操作列只支持“预览”和“导出CSV文件”。
- **未闭环痛点**：业务方花大力气圈出的人群分类，目前无法按照 PRD 要求一键【绑定派发为营销任务事件】。
- **📄 对应 PRD 需求文档条款**：
  > **4.8. 顾客分群管理模块 & 4.9. 商机管理模块**：商机专注分发，基于系统内的标签和分群进行商机规则配置功能。
- **填补建议 (CRUD)**：
  - **触发 (Action)**：针对状态 `Active` 的分群记录，补充极关键的【推送到商机/营销触达】操作按钮，拉起指令弹窗触发服务端投递队列。

### 4️⃣ 档案管理与手工上传：业务可用性断层 (数据上传后未形成流转闭环)
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：系统已开发完整的 `dataQualityWorkbench` (数据质量工作台) 进行多源 (C@P, POAS, WWS) 数据上传与预检，但无法将其拦截的异常数据导向给业务处理人员。
- **📄 对应 PRD 需求文档条款**：
  > **4.1.1. 多源数据接入**：明确包含 C@P, POAS, Voucher, WWS 以及 Manual Files 的 Connect Type 为 `Manual upload to C360`。
- **填补建议 (CRUD)**：
  - **连/改 (Link/U)**：需要在 `dataQualityWorkbench` 的预检异常结果流中补充异常流转派发，直接实现“导入不合格抛出 -> 连线至异常人工校验控制中心 (`errorCorrection`) -> 人工修补属性 -> 正式落库”的连贯审查链路闭环。

### 5️⃣ 审批规则中心：当前版本未实现通用审核能力（需求延期处理）
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：PC 端当前版本没有落地统一的审核/审批模块；早期用于展示的 `approval` 示例页面及相关路由已从产品中移除，系统内所有高危动作（如标签发布、分群导出等）目前均为“直接生效”模式，不存在真正的“待审核/审核中”状态流转。
- **📄 对应 PRD 需求文档条款**：
  > **4.7.2. 标签生命周期管理**：包括标签创建、属性修改、版本控制、废弃停用等操作，确保标签体系的规范性。
- **填补建议 (CRUD)**：
  - **查/改 (R/U)**：务必增配 `ObjectType`（对象判断类型，需包纳 IdentityMerge, TagCreate, SegmentExport 等业务范畴的审核能力展示支撑）。

### 6️⃣ 客户列表核心操作：批量处理、导出与纠错反馈形同虚设（仅前端Mock）
- **严重级别**：🟠 **[高 / High]**
- **当前状态**：在 `customerList` 的批量操作区以及 Profile360 的纠错反馈（Feedback）中，提交操作绝大多数为硬编码的 `console.log` 或弹窗提示 `"功能开发中"`。
- **未闭环痛点**：缺乏真正的修补与出入库流转能力。客服人员发现错误报告纠错后，数据无法真实流转递交；运营人员也无法拿到真实的 Excel 导出文件做离线触达。
- **📄 对应 PRD 需求文档条款**：
  > **2.1.4.1 客户列表**：纠错反馈（提交数据错误反馈）、数据导出。
  > **2.1.4.2 客户批量操作**：批量处理客户数据。
- **填补建议 (CRUD)**：
  - **增/查/改 (C/R/U)**：务必将 `customerList/batchOperation.vue` 及 `index.vue` 中的导出、批量添加和纠错反馈接口接通真实服务端路由，生成实际的流转工单或文件流。

## ✅ 目标端到端生产闭环（标签 → 分群 → 商机 → 跟进）

> 本节为 **目标闭环能力设计**，用于后续立项与评审时对齐“真正可投产闭环系统”应具备的业务与技术逻辑，当前版本仅实现其中部分 UI 和 Mock 能力。

### A. 主业务链路（高层视角）

1. **标签管理（TagManage）**
   - 业务在 `tagManage` 中创建和维护标签，定义人群画像维度（属性类标签、行为类标签等）。
   - 标签有清晰的生命周期：草稿 → 待发布 → 已发布（可用）→ 停用/废弃。
2. **分群管理（Segment / CustomerSegmentation）**
   - 在 `customerSegmentation` 构建分群规则：基于标签 + 属性条件组合，形成可复用的“人群资产”。
   - 分群支持版本控制：每次规则调整都会形成新版本，并记录规则快照。
   - 分群有状态：草稿 / 计算中 / 已生效 / 停用；仅“已生效”版本可被下游商机规则引用。
3. **商机分发规则（LeadRuleConfig）**
   - 在 `leadManagement` 中基于“分群 + 叠加标签 + 时间/事件条件”配置商机分发规则。
   - 规则定义：触发时机（定时 / 事件）、目标系统（BDC、企微等）、去重策略、优先级等。
4. **商机生成与推送**
   - 调度任务按照规则周期性扫描符合条件的客户，生成商机（Lead），写入 C360 商机表。
   - 同步（或异步队列方式）将商机推送到 BDC / 企微 / CRM 等外部系统，并在 C360 中记录推送结果。
5. **跟进与结果回收**
   - BDC / DMS / CRM 等源系统在各自闭环后，将商机状态、成交结果、战败原因等回流至 C360。
   - C360 按 Lead 维度汇总全生命周期：已生成 → 已推送 → 跟进中 → 成交 / 战败 / 失效。
6. **运营分析与策略回溯**
   - 从“标签 / 分群 / 规则 / 商机 / 跟进结果”跨层级打通指标：转化率、ROI、渠道效果等。
   - 支持按“标签版本 / 分群版本 / 规则版本”做 A/B 对比分析。

### B. 标签 → 分群 → 商机 的依赖与状态联动逻辑

#### 1. 标签状态变更对下游的影响

- **依赖关系建模**
  - 系统维护标签与分群/规则之间的依赖关系图：  
    `Tag` ← 被引用于 ← `Segment` ← 被引用于 ← `LeadRule`。
- **当标签被停用 / 废弃时：**
  - 系统应自动扫描所有引用该标签的分群与商机规则，执行：
    - 给出清单：哪些分群 / 规则受影响。
    - 默认策略（可配置）：  
      - 要么直接将相关规则标记为“风险中/待处理”，暂停执行；  
      - 要么要求业务显式确认后才能完成标签停用。
  - 审计上需要记录：是谁在何时停用了标签，对哪些规则产生了影响，方便稽核。

#### 2. 分群状态变更对下游的影响

- **当分群从“已生效”切换为“停用 / 草稿”：**
  - UI 上禁止继续把该分群用作新建规则的来源。
  - 对已存在、依赖该分群的商机规则：
    - 规则进入“待处理/暂停”状态，调度不再实际生成商机。
    - 在规则列表中给予明显的风险标记（如“依赖分群已停用”）。
  - 如业务希望用新版本分群替换旧版本，应支持“一键替换”并触发规则回归测试/小流量发布。

#### 3. 商机规则本身的状态与变更管控

- 规则生命周期：草稿 → 待发布 → 已发布（执行中）→ 暂停 → 归档。
- 任一关键上游对象（标签 / 分群）发生高风险变更时，应提供：
  - 批量“暂停相关规则”的能力；
  - 或通过审批中心先评审变更，再允许规则恢复执行。

### C. 调度与数据流（技术视角）

1. **定时批处理链路**
   - 每日 / 每小时调度任务：
     1. 读取“已发布且启用”的规则集。
     2. 对应分群进行增量或全量计算，获取人群列表。
     3. 结合去重策略生成新的商机列表（避免重复骚扰）。
     4. 将商机写入 C360 的 Lead 表，并打上规则 / 分群 / 标签版本号。
2. **推送链路**
   - 对于状态为“待推送”的商机：
     - 通过接口 / 队列推送至目标系统（BDC、企微等）。
     - 接收推送结果（成功 / 失败 / 重试）并记录在审计日志中。
3. **结果回收链路**
   - 周期性从 BDC / DMS / CRM 拉取最新的商机状态：
     - 更新 C360 Lead 的状态字段（跟进中 / 成交 / 战败 / 失效）。
     - 追加成交金额、战败原因、实际负责顾问等字段。

### D. 端到端生产闭环的业务要求小结

- **从“标签定义”到“分群策略”再到“商机任务”必须是可追溯的：**  
  任一条商机都能反查：是由哪一版标签、哪一版分群、哪一条规则生成。
- **任何上游配置变更必须有下游联动策略：**  
  停用标签 / 分群不会悄悄“让规则继续跑”，而是强制触发风险提示或暂停。
- **闭环的终点是“结果回收 + 指标分析”，不是“推送结束”：**  
  需要从外部系统回收成交/战败结果，在 C360 中形成真正的“闭环看板”。

## 🟡 其他模块级 CRUD 功能健全与交互体验优化（次要改进项）

### 7️⃣ 异常合并控制中心 (Error Correction) 缺少操作动作“后悔药”
- **严重级别**：🟡 **[中 / Medium]**
- **填补建议 (R/D)**：必须在处理界面中独立出来另一个 Tab 专门针对【人历史主导操作台账查阅】，提供并关联【重回初始状态撤销】的回滚命令流转能力。

### 8️⃣ 数据商业看板：偏袒侧重于 IT 与系统运行基准，忽略实际客户商业 ROI 转化
- **严重级别**：🟡 **[中 / Medium]**
- **填补建议 (R)**：建立一个新的菜单节点进入【业务核心成果看盘】，追踪由“新立项 -> 完成派发BDC -> 有效回看 -> 已落成收款”的转化比率数值统计展现。

### 9️⃣ 全局标签管理：与 PRD 所述“审核流程”存在差异（当前版本为“直接生效”设计）
- **严重级别**：🟡 **[中 / Medium]**
- **当前状态**：在 `tagManage` 列表中，新增、编辑、启用、停用标签均是立刻生效操作（直接调用 API），系统中也不存在任何实际运行的审批中心或审核流。
- **📄 对应 PRD 需求文档条款**：
  > **2.1.6 标签管理**：审核流程（审核中状态提示）。
- **填补建议 (R/U)**：修改标签的新增与启用/停用逻辑，提交后状态应变更为 `审核中 (pending)`，并实质性推入到 `approval` 审批中心。

### 🔟 全局标签设定上下架全生命周期缺乏“防误删”屏蔽安全规范
- **严重级别**：🟢 **[低 / Low]**
- **填补建议 (R/U/D)**：仅应当以“不合法隐写逻辑进行失效标记控制（软删除）”，强力阻挡下发于全站各层各类筛选下拉列表中。
