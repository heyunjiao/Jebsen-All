# C360 系统商机模块 (Opportunity) 字段表结构设计（按 Excel 分类重构版）

**文档状态：** 草稿 / 待评审  
**重构依据：** `docs/备份/所有商机字段表整理.xlsx`  
**核心目标：** 严格按 Excel 中的字段分类重构商机字段体系，拆分为 **表 1：全局通用与画像映射字段** + **表 2：场景特有字段（5 个业务域）**。

---

## 1. 设计原则

1. **通用字段入主表。** 任何商机都共用的字段统一进入 `c360_opportunity_master`。
2. **客户与车辆画像不重复落库。** 姓名、性别、联系电话、客户标签、车系车型年款车牌等，默认通过 `vin` 或 `customer_id` 实时映射。
3. **场景差异字段入 JSON 扩展。** 所有业务域特有字段统一落 `ext_attributes`。
4. **先做字段合并，再做模板归属。** 按 Excel 的“合并同类项”口径统一命名，避免不同模板同义字段重复。
5. **状态与时效单独受控。** `opp_status`、`lost_reason`、`task_expire_period`、`expire_date` 在主表统一管理。

---

## 2. 标准存储结构

- **主表：** `c360_opportunity_master`
- **扩展字段：** `ext_attributes`（JSON）
- **扩展 JSON 按 5 个业务域拆分：**
  - `service_time_series`
  - `survey_tickets`
  - `intention_sales`
  - `insurance`
  - `campaign_hook`

## 3. 表 1：全局通用与画像映射字段

> 说明：这些是商机系统底座字段。不管是什么商机，字段逻辑都通用。   
> PM 明确要求：客户和车辆信息不要重复建表存储，直接通过 `vin` 去画像库动态映射。


| 源表字段名  | 标准字段名                | 是否落库 | 落点   | 说明 / 研发规范                                 |
| ------ | -------------------- | ---- | ---- | ----------------------------------------- |
| 车架号    | `vin`                | Y    | 主表   | 核心唯一标识，强校验，全局必填，没有它商机不成立                  |
| 公司编码   | `company_code`       | Y    | 主表   | 商机归属门店，全局必填                               |
| 商机类别   | `business_category`  | Y    | 主表   | 商机大分类                                     |
| 商机类型   | `business_content`   | Y    | 主表   | 细分商机模板名称                                  |
| 商机来源   | `activity_source`    | Y    | 主表   | 自动生成 / 手工导入 / 外部系统同步等                     |
| 商机到期日期 | `activity_due_date`  | Y    | 主表   | 业务要求跟进的最佳时间，核心算法锚点                        |
| 任务失效周期 | `task_expire_period` | Y    | 主表   | 商机失效规则，如 N 天 / N 周；系统据此计算 `expire_date`   |
| 商机失效时间 | `expire_date`        | Y    | 主表   | 到期后系统强制流转为“失效”状态                          |
| 销售顾问编码 | `owner_code`         | Y    | 主表   | 销售跟进人                                     |
| 服务顾问编码 | `owner_code`         | Y    | 主表   | 售后跟进人；与销售顾问编码统一抽象为归属人编码                   |

| 客户类型   | `cust_type`          | N    | 画像映射 | 客户生命周期标签，随画像库更新                           |
| 客户状态   | `cust_state`         | N    | 画像映射 | 客户生命周期标签，随画像库更新                           |
| 行政区    | `region_code`        | Y    | 主表   | 强制转为标准 Code，不直接存中文                        |
| 车系     | `series`             | N    | 画像映射 | 不建字段，根据 VIN 实时查                           |

### 4.1 车辆维保与时序类 `ext_attributes.service_time_series`

**涵盖模板：** 首次保养、首保 2 年、定期保养、定保客户 365 天未保养、车辆 365 天未进店、新车 365 天未保养、新车保修到期、延保到期、保养潜在流失、PSP 到期。


| 源表字段名                         | 标准字段名                       | 处理方式 | 业务说明                  | 适用模板                                            |
| ----------------------------- | --------------------------- | ---- | --------------------- | ----------------------------------------------- |
| 交车日期 / 交付日期                   | `delivery_date`             | 合并保留 | 业务含义一致，统一为交付日期        | 新车交付 90 天回访、新车满意度回访、新转续、续传续                     |
| 本店上一次服务日期 / 上次服务日期 / 上次本店保养日期 | `last_store_service_date`   | 合并保留 | 统一“本店服务”口径            | 车辆 365 天未进店、定保客户 365 天未保养、保养潜在流失、通用商机           |
| 本店上一次服务日期 / 上次服务日期 / 上次本店保养日期 | `last_network_service_date` | 合并保留 | 统一“全网服务”口径，用于风险识别     | 车辆 365 天未进店、定保客户 365 天未保养、保养潜在流失、通用商机           |
| 服务日期 / 进店日期                   | `visit_service_date`        | 合并保留 | 记录本次商机相关的客户到店时间       | 首次保养、定期保养、首保 2 年                                |
| 保修到期日期                        | `warranty_due_date`         | 保留   | 原厂 / 延保口径必须由后端统一输出唯一值 | 首次保养、首保 2 年、定期保养、新车 365 天未保养、新车保修到期、延保到期、保养潜在流失 |
| PSP 到期日期                      | `psp_due_date`              | 保留   | PSP 业务专属时间点           | PSP 到期、首次保养、首保 2 年、定期保养、定保客户 365 天未保养           |

## 6. 结论

基于 Excel 原始分组，当前商机字段结构应固定为：

- **1 张主表**：承载所有通用底座字段与状态控制字段
- **1 个扩展 JSON**：按 5 个业务域拆分特有字段
- **客户与车辆画像字段不重复落库**：统一通过 `vin` / `customer_id` 动态映射
- **字段命名按“合并同类项”后的标准名实施**：避免不同模板重复造字段
