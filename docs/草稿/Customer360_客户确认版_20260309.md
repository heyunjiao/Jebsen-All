## 2. 客户主体设计原则

## 2.1 客户类型识别规则

- 系统当前不单独设计 `企业主体层`、`企业联系人层` 多层对象。
- `公司字段（string）` 有值时，识别为 `公司类型`。
- `公司字段（string）` 无值时，识别为 `个人类型`。
- `客户类型` 属于系统派生字段，来源于 `公司字段（string）` 是否有值。
- 公司类型与个人类型当前核心差异，只有 `公司字段（string）` 是否有值。

## 2.2 购车人/送修人展示规则

- 当前关系只保留两类：`购车人`、`送修人`。
- `购车人` 作为客户主档主体信息展示。
- `送修人` 仅来源于售后订单回传。
- `送修人` 不支持前台修改，只做展示。
- `送修人` 当前只展示两个字段：
  - `送修人姓名`
  - `送修人电话`
- 当前不再设计：
  - `联系人`
  - `经办人`
  - `使用人`
  - `主联系人`
  - `默认送修人`
  - `默认用车人`
- 页面展示建议顺序：
  - `购车人信息`
  - `公司字段（string）`
  - `送修人信息`
  - `关联业务信息`

## 2.3 字段分类说明


| 字段类别   | 对应内容           | 典型字段                          | 归类说明        |
| ------ | -------------- | ----------------------------- | ----------- |
| 主档类    | 客户基础资料与联系方式    | 购车人姓名、联系电话、公司字段、地址、出生日期       | 用于识别客户、联系客户 |
| 关系展示类  | 当前页面仅展示的送修信息   | 送修人姓名、送修人电话                   | 只做展示，不支持维护  |
| 交易事实类  | 已发生的业务事实记录     | 合同号、签单时间、贷款金额、保单号、服务单号、活动参与记录 | 以源业务系统事实为准  |
| 标签/运营类 | 人工维护、运营使用的标记字段 | 客户标签、自定义分组、运营备注               | 用于运营筛选、业务触达 |


说明：

1. `送修人姓名`、`送修人电话` 归为 `关系展示类`。
2. `客户标签`、`人工运营分组` 归为 `标签/运营类`。
3. `电话标签`、`邮箱标签`、`地址标签` 用于区分多条联系方式。
4. `送修人` 信息仅来源于售后订单，不支持在 C360 前台维护。
5. 动态计算字段不纳入本次客户确认范围。

## 2.4 编辑权限说明

本节为建议口径，最终是否允许用户修改，建议由客户确认后定稿。


| 标记      | 建议含义             | 典型字段               |
| ------- | ---------------- | ------------------ |
| `可直接修改` | 用户修改后直接生效，系统保留日志 | 手机号、人工标签、运营备注      |
| `可申请修改` | 用户发起修改申请，审批后生效   | 购车人姓名、公司字段、地址、出生日期 |
| `仅展示`   | 页面只读展示，不支持前台修改   | 送修人姓名、送修人电话        |
| `不可修改`  | 只读展示，以源系统事实为准    | 合同号、金额、签单时间、放款信息   |

## 4.1 上传处理规则

1. 上传必须基于标准模板。
2. 上传前先校验模板版本、表头、必填字段、字典值和主键。
3. 上传采用 `有错整批退回`，不建议允许“部分成功”。
4. 每次上传必须生成批次号并保留回执。
5. 每次失败必须提供失败明细下载。
6. `预校验失败`、`入库校验失败`、`无权限上传` 统一在上传工作台闭环，不进入异常中心。
7. 上传时系统使用 `BPID`、`手机号`、`证件号`、`VIN` 挂接 `C360` 现有主体；若命中多个有效 `OneID` 且无法识别唯一挂接对象，按上传错误处理。
8. 上传时若记录与现有 `C360` 档案存在相似关系，但系统无法确认是否为同一主体，则只展示候选档案、相似度分值、命中规则，不自动合并 `OneID`。
9. 若该记录后续仍被允许入库，系统需在异常中心生成 `EX-UNI-010 相似用户 OneID` 异常任务；异常详情只展示候选档案、相似度分值、命中规则，供人工判断是否合并、拆分或保留，系统不自动执行任何合并动作。

### 5.1.3 进入边界与前置依赖

边界规则：

1. 异常中心不接收 `手工上传被拒绝的文件`、`手工上传预校验失败批次`、`手工上传入库校验失败批次`。
2. 异常中心只处理 `已入库后的存量数据异常`、`用户反馈纠错异常`。
3. 手工标签只允许在 `H5` 和 `C360后台` 的 `设置新增/编辑` 中圈选、取消；`BDC`、`WeCom` 只读；标签维护动作不生成异常任务，只保留操作日志。
4. 标签数据质量异常仍进入异常中心，按 `有效性异常`、`唯一性异常`、`完整性异常`、`一致性异常` 治理。

### 5.1.4 异常中心设计的前置依赖


| 前置依赖      | 必须先确认的内容                                                                     | 不确认的影响                       |

### 5.1.5 按字段修改边界判断是否进入异常中心


| 字段修改边界  | 处理原则                                  | 是否进入异常中心 |
| ------- | ------------------------------------- | -------- |
| `可直接修改` | 前台直接修改并保留审计日志；回写失败由接口重试和运维日志闭环        | 默认不进入    |
| `可申请修改` | 用户提交修改申请；审批流、驳回、回写异常统一纳入异常中心追踪        | 进入       |
| `仅展示`   | 只读展示，以源系统同步结果为准；如送修人信息有误，只能通过源系统修正后同步 | 默认不进入    |
| `不可修改`  | 不允许前台改值；只能通过异常中心核查、指定信任源、审批决议处理       | 进入       |

| `EX-VAL-009` | 标签编码无效   | 已入库标签记录中的 `标签编码` 不在 `C360` 已发布手工标签配置内，或该标签已停用 |


### 5.3.3 唯一性异常


| 错误码          | 场景          | 异常出现方式                                                                                                                                                                             |
| ------------ | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `EX-UNI-001` | OneID 冲突    | 系统初始化、源系统同步、上传入库、实时接入后，`C360` 基于挂接规则、去重规则为同一自然人生成了多个有效 `OneID`；冲突由 `C360` 内部生成                                                                                                     |
| `EX-UNI-002` | 同联系电话多个客户   | 同一联系电话同时绑定多个客户档案，且姓名、证件、地址无法证明为同一自然人；包括销售顾问电话误填到多个客户、家庭成员共用电话、购车人电话混填                                                                                                              |
| `EX-UNI-003` | 同证件号多个客户    | 同一证件号对应多个客户档案                                                                                                                                                                      |
| `EX-UNI-004` | 同 VIN 多主体冲突 | 同一 VIN 同时挂接多个有效客户主体                                                                                                                                                                |
| `EX-UNI-008` | 同 BPID 多主体  | 同一 `BPID` 同时挂接多个有效客户主体                                                                                                                                                             |
| `EX-UNI-005` | 同合同号重复      | 同一合同号对应多条销售事实记录                                                                                                                                                                    |
| `EX-UNI-006` | 同保单号重复      | 同一保单号对应多条保险事实记录                                                                                                                                                                    |
| `EX-UNI-007` | 同券号重复       | 同一券号对应多条权益记录                                                                                                                                                                       |
| `EX-UNI-009` | 同一对象重复标签    | 同一主体被重复挂接相同 `标签编码`，形成多条有效标签记录                                                                                                                                                      |
| `EX-UNI-010` | 相似用户 OneID  | 系统初始化、源系统同步、上传入库、实时接入、批量巡检时，`C360` 按姓名、联系电话、地址、BPID、WeCom EID、VIN 计算相似度；当新记录与已有 `OneID` 的相似度达到提示阈值但未达到自动合并阈值时，只展示候选档案、相似度分值、命中规则，生成相似用户 `OneID` 异常任务，供人工判断是否合并、拆分或保留，系统不自动执行任何合并动作 |

### 5.3.5 一致性异常


| 错误码          | 场景             | 异常出现方式                                                       |
| ------------ | -------------- | ------------------------------------------------------------ |
| `EX-CON-001` | 公司字段与客户类型识别冲突  | `公司字段（string）` 有值，但页面识别为个人；或 `公司字段` 无值，但页面识别为公司              |
